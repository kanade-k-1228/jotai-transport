# 計測結果（baseline / after）

計測環境: Node v22.19.0 / Linux WSL2 / `jotai@2.x`。マイクロベンチ（vitest bench）はこのマシンでは
分散が大きく（rme ±5〜14%、送信経路など未変更コードでも run 間で 20%超ぶれる）小差の判断には使えない。
**受信ホットパスの安定指標は `profile.ts`（長時間反復・3回平均）を用いる。**

> 注: 下記の baseline / after の数値は、最適化を入れた当時（受信が毎メッセージ同期ディスパッチ）の計測。
> その後、受信は常にコアレッシング（microtask 単位でまとめ最新値だけ配信）へ一本化したため、
> `profile.ts` / `micro.bench.ts` は emit ごとに microtask を回して 1 件ずつ flush させる形に更新済み。
> A〜C の最適化は coalesce flush 側のディスパッチにもそのまま効いている。

最適化（`pkgs/jotai-transport/src/index.ts`）の内容:

- A. 受信ループの `Object.entries` → `for...in`（毎メッセージの `[k,v]` タプル配列確保を排除）
- B. リスナ配信の `Set.forEach((fn)=>…)` → `for...of`（配信毎のクロージャ確保を排除）
- C. `setSelf` を直接リスナとして渡す（配信毎のラッパ呼び出しを排除）
- 値デデュープは**不採用**（jotai が `Object.is` で同値書き込みを短絡しており効果が薄い）
- `whenReady` の即時購読の遅延化は**見送り**（受信ホットパスに無関係・初期値解決タイミングの意味論変更を伴う／
  E2E ヒープも per-client ≈0.2MB と小さい）

## 受信スループット（profile.ts, プロファイラ無し, 2M×2キー, dispatch/s）

|           | run1    | run2    | run3    | 平均         |
| --------- | ------- | ------- | ------- | ------------ |
| baseline  | 115,392 | 110,072 | 115,159 | ≈ 113,500    |
| **after** | 126,747 | 134,622 | 125,684 | **≈ 129,000** |

→ **+13.6%**（before/after でレンジが重ならず、ノイズではなく実効）。

## CPU プロファイル（受信ホットパス, 2キー更新 ×3M）

self-time の**割合**と**絶対値**（割合は分母＝総サンプルに対する比。自前コードを削ると jotai/GC の割合は
相対的に上がって見えるので、絶対値とフレームの有無で判断する）:

| frame                         | baseline 割合 | after 割合 | 備考                          |
| ----------------------------- | ------------- | ---------- | ----------------------------- |
| jotai `readAtomState`         | 23.2%         | 24.0%      | 派生 atom 再計算（不可避）    |
| (garbage collector)           | 16.8%         | 18.1%      | 下記の絶対値参照              |
| jotai `recomputeInvalidated…` | 9.6%          | 10.6%      | 派生 atom 再計算（不可避）    |
| jotai `flushCallbacks`        | 6.2%          | 6.7%       | 不可避                        |
| `parseMessage`（lib）         | 5.5%          | 5.5%       | JSON.parse 主体・不可避       |
| **`ws.onmessage`（lib）**     | **4.7%**      | **≈0%**    | **上位から消滅（最適化対象）** |
| リスナ/forEach/派生クロージャ | 約 3.3%       | ≈0%        | 上位から消滅                  |

絶対値（同一 3M dispatch）:

| 指標                       | baseline  | after     | 差     |
| -------------------------- | --------- | --------- | ------ |
| 総 CPU 時間（profiler 下） | 31,735 ms | 26,907 ms | -15.2% |
| GC サンプル数              | 4,626     | 4,230     | -8.6%  |
| dispatch/s（profiler 下）  | 94,534    | 111,495   | +17.9% |

→ 自前の受信コード（`onmessage` の `Object.entries`+`forEach`+クロージャ ≈ 8%）がプロファイル上位から消え、
総 CPU と GC の**絶対量**が減少。残りはほぼ jotai 本体（派生 atom 再計算）で、これは公開 API 構造上ほぼ不可避。

## E2E 負荷（50 clients / 2000 updates / 8 keys, `--expose-gc`）

|                                   | baseline                 | after                    |
| --------------------------------- | ------------------------ | ------------------------ |
| ingest throughput                 | 23,868 msg/s             | 25,309 msg/s（+6%）      |
| fan-out latency p50 / p99 / max   | 162.0 / 442.2 / 513.1 ms | 158.5 / 488.8 / 542.7 ms |
| client heap baseline / per-client | 12.0 MB / ≈0.2 MB        | 11.9 MB / ≈0.2 MB        |

E2E はサーバのファンアウト＋ネットワーク律速のため、クライアント CPU 改善の寄与は小さい（throughput +6%、
遅延 p99 は run 間ノイズ）。クライアントが CPU 律速になる純粋な受信ディスパッチでは +13.6%（上記）。

## 受信のコアレッシング採用根拠（履歴）

下記は、受信を「同期で全件配信」する版と「microtask でキー毎に最新値だけ配信」する版を比較した実験の記録
（当時の `src/coalesce-experiment.ts`、現在は削除）。この結果を踏まえ、コアレッシングを**唯一の挙動**として
採用した（同期版・opt-out は撤去）。同一ティックに BURST 件届くバーストを ROUNDS 回流して比較。
CPU 律速にするため await 回数を抑えた構成も併記。

| rounds × burst | sync 通知数 | coalesced 通知数 | sync msg/s | coalesced msg/s | 速度比 |
| -------------- | ----------- | ---------------- | ---------- | --------------- | ------ |
| 2000 × 1       | 2,001       | 2,000            | 818        | 875             | 1.07x  |
| 2000 × 10      | 20,001      | 2,000            | 8,309      | 8,677           | 1.04x  |
| 2000 × 50      | 100,001     | 2,000            | 36,731     | 43,734          | 1.19x  |
| 2000 × 200     | 400,001     | 2,000            | 90,992     | 154,917         | 1.70x  |
| 200 × 2000     | 400,001     | 200              | 226,637    | 1,135,783       | 5.01x  |
| 100 × 5000     | 500,001     | 100              | 203,610    | 1,854,453       | 9.11x  |

（2000×… は await×2000 の床に律速され速度比が圧縮される。…×2000/5000 が純 CPU 差に近い）

結論:

- **通知回数（≒ React 再描画）はバースト係数ぶん削減**（決定論的。BURST=200 なら 200x、5000 なら 5000x）。
- **処理スループットはバーストが大きいほど伸びる**（同一ティック多数更新で 5〜9x）。jotai の per-write
  再計算（プロファイル上 ~60%）を「キー毎に1回」へ畳むため。
- **バースト無し（1 msg/tick）では利得ほぼ無し**で、microtask 1 ティック分の遅延だけ増える。
- **代償: 中間値が配信されない**（キー毎に最新値のみ）。状態同期用途では「最後の値が現在値」で十分なため、
  この挙動を唯一の仕様として採用した（毎更新を 1 件ずつ数える用途には不向き）。
